# P4-2 — 2초 영상 신호 read-only 설계 감사

- 작성일: 2026-07-05
- 기준 커밋: main / 93df86e (P4-1 시리즈 기능 마감 시점)
- 성격: read-only 감사 문서. 이 문서 외 코드 변경 없음. **구현 금지 단계** (camera permission, getUserMedia, MediaRecorder video, video upload, schema/migration, Storage 변경 전부 미실행).
- 기능명: **2초 영상 신호** (P4-0/P4-1B의 2초 음성 신호와 같은 "짧은 관계 신호" 원칙을 영상에도 적용할지 판단하는 문서)

## 0. 참고 — 2초 음성 신호(P4-1) 구현 현황 요약

영상 설계는 이미 실전송 중인 음성 구조를 기준선으로 삼는다.

| 항목 | 음성(P4-1B) 실제 값 |
|---|---|
| 용량 상한 | 300KB (`MAX_SIZE_BYTES = 307200`) |
| 길이 상한 | 2.5초 (`MAX_DURATION_MS = 2500`) |
| 허용 MIME | `audio/webm`, `audio/mp4` (Android=webm, iOS Safari=mp4 fallback) |
| 만료 | 24시간 (`EXPIRES_MS`), 72h 검토 후 대장 최종 확정 |
| Storage | private bucket `voice-signals`, signed URL만 |
| signed URL TTL | 120초 (`SIGNED_URL_TTL_SECONDS`) |
| 연결 검증 | 서버 `findConnection` — `dl_invites` accepted + 세션 legacy id 집합 |
| DB 제약 | `signals` 테이블 `type CHECK IN ('emoji','voice')` + voice일 때 `audio_path/audio_mime/audio_duration_ms(1~2500)/audio_size_bytes(1~307200)/expires_at` 전부 NOT NULL 강제 |
| cleanup | `type='voice' AND expires_at < now()`, storage 삭제 성공 배치만 row 삭제(멱등), batch 100/limit 1000 |
| Point | `signal-days` 쿼리가 `type` 필터 없이 `sender_id`만 필터 → voice도 자동으로 signalDays에 포함, 별도 보너스 없음(P4-1E 확정) |
| 알려진 미검증 리스크 | iOS Safari 실기기 미검증, 카카오 인앱 브라우저 미검증(Android Chrome만 실기기 PASS) |

이 표의 마지막 줄이 이번 감사의 출발점이다 — **음성조차 iOS/카카오인앱이 미검증인 상태에서 영상을 얹는 것의 리스크**를 먼저 따진다.

## 1. 제품 기준

- **"카톡은 대화, 던바링크는 2초 신호"에 영상이 맞는가**: 부분적으로만 맞는다. 이모지→음성까지는 부담이 완만하게 늘지만(문자 한 개 → 목소리 2초), 영상은 촬영 시 얼굴·배경·조명·옷차림을 신경 쓰게 되는 심리적 부담이 계단식으로 커진다. "짧은 신호"가 아니라 "미니 콘텐츠 제작"으로 체감이 바뀔 위험이 있다.
- **영상이 음성보다 실제 가치 있는 순간**: 표정으로만 전달되는 뉘앙스(장난스러운 표정, 순간의 리액션)는 음성보다 영상이 낫다. 다만 이런 "특별한 순간"은 카카오톡 영상통화/일반 메신저로도 충분히 처리 가능한 영역이라, 굳이 이 앱에 얹어야 할 필연성이 약하다.
- **부담/피로/사생활 리스크**: 얼굴이 고정 노출되는 매체라 음성(목소리만)보다 개인정보 민감도가 명확히 높다. 배경에 타인·사생활 공간이 찍힐 위험도 음성엔 없던 리스크다.
- **MVP에 꼭 필요한가**: 아니다. 음성 신호 자체가 이제 막 실기기 확인(P4-1C~e)을 마쳤을 뿐 실사용자 데이터가 전무하다(테스트 계정은 대장 본인 2개뿐). 검증되지 않은 음성 위에 리스크가 더 큰 영상을 얹는 것은 순서상 이르다.

## 2. UX 기준

- **영상 버튼 상태**: 현재 `person-detail-client.tsx`의 `handleVideoNotice()`가 "2초 영상 신호는 다음 단계에서 열려요."를 띄우는 준비중 안내만 존재(`two-second-room.tsx` 우선순위 주석: "🎙️ 2초 음성(메인) > 🎥 2초 영상(준비중, P4-2) > 😊 이모지(보조)"). **이 상태를 유지 추천** — 아래 3~6절 리스크가 해소되기 전까지.
- **구현 시 흐름**: 촬영 → 미리보기 → 다시 촬영 → 보내기. `voice-signal-preview.tsx`의 상태 머신(녹음중/미리듣기/재녹음/전송중/실패)을 그대로 video로 확장 가능한 구조 — MediaRecorder 스트림에 video 트랙만 추가하는 정도로 UI 골격은 재사용 가능.
- **자동재생 금지**: 필수. 현재 음성도 `<audio controls>`로 자동재생 없음 — 영상은 더 엄격히 지켜야 한다(예상치 못한 순간에 얼굴 영상이 재생되면 노출 부담이 음성과 비교 불가).
- **다운로드/공유 버튼 금지**: 필수, 현재 음성도 `controlsList="nodownload"`로 다운로드 버튼을 숨긴다. **단, 이 한계를 문서로 명확히 해야 한다**: `nodownload`는 브라우저 기본 UI의 다운로드 버튼만 숨기는 것이고 화면 녹화/캡처까지 막지는 못한다. "안전하게 사라진다"는 과신을 UX 문구에 심지 않아야 한다.
- **24시간 만료**: 음성과 동일 원칙 유지 권장. 영상이 더 민감하다는 이유로 단축(예: 12시간)도 검토 가능하나, 이는 제품 판단이지 기술 제약은 아니다.
- **1회 재생 후 숨김**: 음성도 도입하지 않은 UX(만료 전까지 재생 제한 없음). 영상도 동일하게 미도입 권장 — 화면 캡처를 막을 수 없는 이상 "1회 재생"은 실질적 보안 효과 없이 복잡도와 "안전하다는 착각"만 늘린다.
- **수신자 부담 경감**: 자동재생 금지 + 명시적 "▶ 영상 보기" 클릭 후에만 노출(음성의 "▶ 음성 듣기" 패턴과 동일) + 향후 필요 시 "받는 사람도 조용한 공간에서 볼 수 있게" 안내 문구 검토.

## 3. 브라우저/기기 리스크

- **Android Chrome**: MediaRecorder video(webm/vp8·vp9 + opus) 지원 안정적. 음성에서 이미 Android 실기기 PASS 확인됨(레퍼런스 있음) — 상대적으로 리스크 낮음.
- **iOS Safari**: 핵심 리스크. iOS Safari의 MediaRecorder video 지원은 상대적으로 늦게(iOS 14.3+) 부분 지원됐고 코덱(H.264/mp4)·안정성·발열 이슈가 Android 대비 예측하기 어렵다. **음성조차 iOS 실기기 미검증인 상태**(known-unverified, blocking 아님으로 분류돼 있음)에서 영상까지 올리면 미검증 표면이 배로 늘어난다.
- **PWA/홈화면 추가 상태**: iOS PWA(standalone) 모드에서 카메라/마이크 권한 프롬프트가 사파리 탭과 다르게 동작하는 사례가 보고돼 있다(권한을 매번 다시 묻거나, 특정 iOS 버전에서 카메라 스트림이 멈추는 경우). 검증 없이 내보내면 P4-1C-e에서 겪었던 "권한 문제인지 코드 문제인지 구분 안 되는" 문제가 카메라에서도 재현될 위험이 있다.
- **카카오 인앱 브라우저**: 음성도 미검증인 채로 남아 있는데, 카카오 인앱은 카메라 권한 자체를 별도 체계로 처리하거나 아예 막는 경우가 있다. 이 앱은 초대 링크를 카카오톡으로 공유받아 인앱으로 들어오는 흐름이 흔해서, 영상 기능이 인앱에서 통째로 안 되면 사용자 경험상 타격이 크다.
- **권한 거부 시 UX**: 음성과 동일 패턴 — 버튼을 완전히 숨기기보다 "카메라를 지원하지 않아요"/"카메라 권한이 필요해요" 안내로 대체.
- **백그라운드/잠금화면/저전력모드**: 2초 촬영이라 절대적 리스크는 낮지만, 저전력 모드에서 카메라 프리뷰 프레임레이트 저하, 촬영 중 앱 전환 시 스트림 강제 종료 처리(에러 핸들링)는 필요.
- **PC Chrome**: 웹캠이 있으면 지원 자체는 원활. 다만 노트북 내장캠 화질/조명 이슈로 "2초 신호"치고 준비 부담이 실제로는 꽤 큼(옷매무새/조명 신경).

## 4. 포맷/용량 리스크

- **예상 용량**: 저해상도(360~480p) 2초 영상 기준으로도 음성(300KB)보다 확연히 크다. 코덱/해상도/비트레이트에 따라 편차가 크지만 대략 500KB~2MB 범위로 추정 — **실측 필요**(현재는 가정치).
- **webm/mp4/mov 호환성**: Android Chrome은 webm(vp8/vp9+opus), iOS Safari는 MediaRecorder로 사실상 mp4(H.264+AAC)만 지원. 음성과 같은 패턴(webm↔mp4 fallback)이지만 video는 비디오+오디오 트랙 조합이라 코덱 매트릭스가 더 복잡하다.
- **서버 변환 없이 원본 그대로 쓸 때 문제**: 브라우저별로 다른 컨테이너(webm vs mp4)가 그대로 저장되면, 재생 측 브라우저가 그 포맷을 지원 못해 "상대가 보낸 영상이 안 열림" 문제가 생길 수 있다. 오디오는 크로스 재생 지원 폭이 넓어 음성에서는 실제 문제가 안 됐지만, video는 브라우저별 codec 지원 매트릭스가 훨씬 좁아 **트랜스코딩 없는 MVP의 가장 큰 걸림돌**이다.
- **썸네일 필요 여부**: 신호함에서 재생 전 정지 이미지 미리보기가 있으면 UX가 좋지만 서버 캡처/트랜스코딩 인프라가 필요하다. 없이도 MVP는 가능(음성처럼 "▶ 영상 보기" 텍스트 버튼만으로 축소) — **권장**.
- **압축/트랜스코딩 없는 MVP 가능 여부**: 가능은 하나 "제한적 MVP"로만 봐야 한다. Android 한정 실험이면 크로스 재생 문제가 발생할 여지가 적어 문제되지 않지만, 전체 브라우저 지원을 전제하면 실사용에서 재생 실패를 자주 만날 확률이 높다.

## 5. Storage/비용

- **150명 기준 하루 사용량 가정**: 현재 실사용자는 대장 테스트 계정 2개뿐이라 150명 전원 활성 가정은 비현실적이다. 참고 가정치로만: 활성 유저 N명이 하루 평균 1.5회 영상 전송, 회당 평균 1MB 가정 시 하루 누적 = N × 1.5MB.
- **24시간 만료 전 최대 저장량**: 만료 전까지는 그날 전송량 전체가 항상 Storage에 남아 있다(음성과 동일 구조). 영상은 회당 용량이 커서 같은 사용자 수 기준으로도 누적 저장량이 음성보다 몇 배 빠르게 는다.
- **Supabase Storage 비용/대역폭**: Storage 용량뿐 아니라 signed URL 재생마다 egress(대역폭) 비용이 발생한다. 영상은 음성 대비 1회 재생당 대역폭 소모가 몇 배 크므로, 사용자 수가 늘면 비용 곡선이 음성보다 훨씬 가파르게 증가한다.
- **signed URL TTL 유지 가능 여부**: 120초 그대로 유지 가능(재생 길이가 2초라 TTL 자체는 문제 없음, 다만 네트워크 느린 환경에서 로딩 여유는 고려).
- **cleanup 확장 시 주의점**: 현재 cleanup 스크립트는 `type='voice'`로 좁게 필터링돼 있다(`.eq("type", "voice")`). video를 추가하려면 `type IN ('voice','video')`로 조건을 넓히고, bucket도 분리(`video-signals` 신설)할지 통합할지 결정해야 한다. **이 문서에서는 설계 언급만 하고 실제 코드는 건드리지 않는다.**

## 6. 보안/개인정보

- **private bucket + signed URL만**: 음성과 동일 원칙, 영상은 얼굴 식별 가능한 생체정보라 더 엄격히 지켜야 한다.
- **public URL 금지**: 동일 유지.
- **다운로드/공유 버튼 금지**: 동일 유지, 단 "완전 차단 불가"(화면 캡처/녹화는 막을 수 없음) 한계를 사용자 안내 문구에 과장하지 않아야 한다.
- **sender/receiver 외 접근 금지**: 음성과 동일 로직(`findConnection` + signed URL 발급 시 `myIds` 검증) 그대로 재사용 가능한 구조.
- **만료 후 삭제**: 동일 원칙, cleanup 확장 필요(5절 참고, 이번엔 미실행).
- **로그 노출 금지**: raw path/URL을 로그에 남기지 않는 기존 원칙 유지.
- **별도 동의/주의 문구 필요 여부**: 필요할 가능성이 높다. 얼굴 노출은 목소리보다 식별성이 훨씬 높고 배경에 타인이나 사생활 공간이 찍힐 위험도 있다. 최초 카메라 권한 요청 시점에 "이 영상은 24시간 후 자동 삭제되고 상대만 볼 수 있어요" 같은 명시적 고지가 법적 요구라기보다 신뢰 구축 차원에서 필요할 수 있다.

## 7. DB/API 설계 초안 (구현 금지 — 설계만)

- **signals 테이블 확장 여부**: 새 테이블 대신 기존 `signals` 재사용(voice와 동일 패턴). `type` CHECK 제약을 `IN ('emoji','voice','video')`로 확장하는 안.
- **컬럼 재사용 vs 일반화**: `audio_*` 재사용보다 **명시적 `video_*` 컬럼 신설을 권고** — 이름 충돌 방지, 추후 해상도/코덱 등 video 특유 메타데이터를 붙일 여지. 후보: `video_path`, `video_mime`, `video_duration_ms`, `video_size_bytes`(모두 nullable, voice와 동일하게 `type='video'`일 때만 NOT NULL을 강제하는 CHECK 제약).
- **기존 audio_\* 컬럼과 충돌 가능성**: 없음(별도 네임스페이스). 다만 한 row에 `audio_*`와 `video_*`가 동시에 채워지는 상태를 CHECK 제약으로 명시적으로 막아야 한다(type이 배타적이라 자연히 방지되지만 제약으로 강제하는 게 안전).
- **API route 설계 후보**: `POST /api/signals/video` — voice route와 대칭 구조(FormData: video+receiverId+durationMs, `findConnection` 재사용, 용량 상한은 음성의 300KB가 아니라 영상 실측 후 재산정 — 예비 후보 2~3MB, 확정 전 실측 필수).
- **signed URL route 후보**: `GET /api/signals/video-url` — voice-url과 대칭, TTL 120초 유지 가능.
- **cleanup 확장 후보**: 기존 스크립트의 `.eq("type", "voice")`를 `.in("type", ["voice", "video"])`로, bucket 순회도 배열화(voice-signals + video-signals 각각 삭제) — 이번 문서에서는 방향만 기록, 실제 수정 없음.

## 8. Point 정책

- **video도 signalDays에 포함할지**: 그렇다. 기존 `/api/me/signal-days` 쿼리가 `type` 필터 없이 `sender_id`만 필터하므로, video row를 `signals`에 넣기만 하면 **코드 변경 없이 자동으로** signalDays에 포함된다.
- **video 별도 보너스**: 금지. P4-1E에서 확정한 원칙(포인트 인플레이션 방지, 남발 유인 차단, "관계 유지 행동" 본질 유지)을 그대로 확장한다. 오히려 촬영 부담이 크다는 이유로 보상을 더 준다면 "부담스러운 촬영을 포인트 때문에 남발"하는 유인이 생겨 정책 취지에 역행한다.
- **emoji/voice/video 같은 날 중복 방지**: 기존 KST 날짜 `Set` dedupe 로직이 `type` 무관하게 동작하므로 자동으로 커버된다(코드 변경 불필요).
- **P4-1E와 일관성**: 완전히 일관 — 같은 원칙의 확장일 뿐 새로운 정책이 아니다.

## 9. 구현 후보안 비교

### A안 — 영상 계속 보류
- 장점: 리스크 0, 개발/운영 비용 0, 음성 사용성 데이터 축적 우선 가능, "준비중" UX가 이미 있어 자연스럽다.
- 단점: 제품 로드맵상 "2초 영상"이 원래 비전의 일부인데 계속 미뤄진다(대장 판단 필요), 사용자 기대 대비 지연.
- 개발 난이도: 없음. 운영 비용: 없음. 개인정보 리스크: 없음.
- **추천/비추천: 추천(현재 시점).**

### B안 — Android Chrome 한정 실험(P4-2A)
- 장점: iOS Safari/카카오 인앱 리스크를 우회, 낮은 리스크로 실사용 데이터·버그 수집 가능, "Android-first" 검증 패턴이 이미 조직 내 선례 있음(P4-1A2 음성 Android 실기기 검증).
- 단점: 대장의 실기기 조합(PC+카카오 모바일)에서 실제 안드로이드 실기기 검증은 가능하지만 iOS 사용자에게는 기능 자체가 안 보여 혼란 가능(기기 분기 UX 별도 필요), 개발 범위가 음성보다 넓다(코덱/화면 방향/전면카메라 기본값 등 video 특유 이슈).
- 개발 난이도: 중. 운영 비용: 중(Android 한정이라 완만하지만 음성보다는 큼). 개인정보 리스크: 중~높음(얼굴 노출, 다만 private+signed URL+24h 만료 유지 시 관리 가능).
- **추천/비추천: 조건부 추천 — 아래 10절 선결조건 충족 후.**

### C안 — 전체 브라우저 지원 전제 구현
- 장점: 완전한 기능, 사용자 차별 없음.
- 단점: iOS Safari/카카오인앱 리스크가 그대로 노출, 크로스 포맷 재생 실패 가능성 높음(4절), 코덱 매트릭스 전체 QA가 필요해 개발/검증 범위가 큼. 음성조차 iOS 미검증인 시점에 영상 전체 지원을 약속하는 것은 과설계에 가깝다.
- 개발 난이도: 높음. 운영 비용: 높음. 개인정보 리스크: 높음.
- **추천/비추천: 비추천(지금 시점).**

### D안 — 영상 대신 사진/짧은 GIF/스티커 등 대체안
- 장점: 촬영 부담이 영상보다 훨씬 낮음(정지 순간 캡처면 충분히 신호 전달 가능), 크로스 브라우저 호환성 문제가 사실상 없음(이미지는 코덱 이슈 없음), 개발 난이도 낮음(getUserMedia+canvas capture, MediaRecorder 불필요), Point/보안/만료 정책은 음성 패턴 그대로 재사용 가능, Storage 비용도 영상보다 훨씬 작음.
- 단점: "2초"라는 시간 개념의 컨셉과는 어긋난다(정지 사진엔 "2초"가 없음), 원래 제품 비전(영상)과 다른 타협이라 대장의 방향성 판단이 필요.
- 개발 난이도: 낮음. 운영 비용: 낮음. 개인정보 리스크: 중(여전히 얼굴 노출이지만 영상보다 노출 정보량·시간이 적음).
- **추천/비추천: 검토 후보로 제시(채택 여부는 제품 방향 판단 — 이 감사의 결론 사항은 아님).**

## 10. 최종 권고

**지금 구현 금지, 음성 사용성을 더 관찰한다.**

이유:
- 음성 신호 자체가 P4-1 시리즈로 방금 기능 마감됐을 뿐(2026-07-05), 실사용자 데이터가 전혀 없다(대장 본인 테스트 계정 2개가 전부).
- 영상은 음성보다 (1) 개인정보 민감도, (2) 브라우저 호환성 리스크(특히 iOS Safari·카카오 인앱), (3) 용량/비용이 전부 더 크다.
- "관계 유지 행동"이라는 본질 유지, 포인트 인플레이션 방지, 과설계 방지라는 제품 원칙에 비춰봐도, 검증되지 않은 음성 위에 더 무거운 영상을 얹을 근거가 아직 부족하다.

**영상을 하게 된다면 선결조건**:
1. 음성 신호 실사용 데이터 확보(최소 1~2주 실사용, 전송 빈도·리텐션 확인).
2. 음성 자체의 iOS Safari·카카오 인앱 미검증 리스크를 먼저 해소(현재 known-unverified 상태 — 이게 해결 안 된 채 영상을 얹으면 미검증 표면이 배로 늘어남).
3. 영상 용량 실측(가정치가 아니라 실제 녹화 파일 크기로 Storage 비용 재계산).
4. 영상을 시작한다면 **B안(Android 한정 실험, P4-2A)**부터 — C안(전체 브라우저 지원)은 당분간 금지.
